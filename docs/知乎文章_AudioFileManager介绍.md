# 我写了个桌面软件，把我的 NAS 里那堆"散装音乐"治得服服帖帖

## 一个中年数码爱好者的真实困境

事情要从年初说起。

我的 NAS 上躺着一个叫 `Music` 的文件夹，里头大概 300 多 G 的音频文件，攒到现在，跨了十五六年。每次想听点啥，打开文件夹，画面大概是这样的：

```
Music/
├── 周杰伦/
│   ├── 01 晴天.mp3
│   ├── 02 周杰伦 - 止战之殇.mp3
│   ├── 兰亭序.flac
│   ├── Jay - 星晴.mp3
│   └── ...（60 多首，命名风格完全随缘）
├── 陈奕迅 - 富士山下.mp3
├── 陈奕迅 - 富士山下.flac（同一首歌两个格式？）
├── 张学友 - 等你等到我心痛.mp3（没有封面，也没有专辑信息）
├── Unknown Artist/
│   ├── 01 不知道谁唱的.mp3
│   └── ...（十几首"来历不明"的文件）
└── 乱七八糟/
    ├── 周传雄-寂寞沙洲冷-酷狗.mp3
    └── ...
```

没有专辑封面、歌手标签乱码、文件名各自为政——Windows 自带的 Groove Music 根本认不全，媒体服务器（Plex/Jellyfin）扫出来的元数据也是一团糟。你也试过手动一个个改，但改了 20 首就放弃了——谁有那个耐心？

市面上不是没有工具。MusicBrainz Picard、MP3Tag、Kid3…但它们要么太丑，要么上手门槛高，要么对中文歌曲的支持很差（毕竟这些工具大多面向欧美用户，iTunes 搜不到的歌就歇菜了）。

所以我决定自己写一个。

> 项目地址：[AudioFileManager](https://github.com/your-repo/AudioFileManager)（开源，Python 编写，纯桌面应用）

这篇文章就是它的完整介绍，我尽量写得干一点，不掺水。

## 这个软件到底能干什么？

一句话总结：

**把一堆"散装"的音乐文件丢进去，它自动帮你搜全元数据、补封面歌词、整理目录——出来的就是一份可以直接进媒体库的"精装"音乐集合。**

具体来说，一条完整的处理流水线：

```
拖入文件夹 → 递归扫描 → 文件名自动解析 → 
多源并行搜索元数据 → 下载封面 & LRC 歌词 → 
统一编码（解决中文乱码）→ 写入 ID3 标签 → 
按「歌手 → 专辑(年份) → 歌曲」自动归类整理
```

这里面每一步都有细粒度控制，后面慢慢说。

## 支持的音频格式

不只是 MP3。这个项目有非常认真的格式覆盖：

| 格式 | 标签系统 | 封面写入 | 歌词写入 |
|------|---------|---------|---------|
| MP3 | ID3v2.4 | ✅ | ✅ |
| FLAC | Vorbis Comment | ✅ | ✅ |
| M4A (AAC/ALAC) | iTunes MP4 | ✅ | ✅ |
| OGG (Vorbis) | Vorbis Comment | ✅ | ✅ |
| WMA | ASF | ❌ | ✅ |
| APE | APEv2 | ✅ | ✅ |

> WMA 不支持封面是因为 ASF 格式的封面写入在 mutagen 里坑比较多，日常也不常用，就跳过了。

## 搜索能力：它是怎么"找到"你的歌的？

这是整个软件最核心的部分。它同时接入 **5 个数据源**，并行搜索、自动降级、相似度匹配：

```
iTunes Search API    → 曲库最大，英文歌首选
MusicBrainz          → 开源元数据权威，严格限速（1请求/秒）
LRCLIB               → LRC 歌词专用源，支持同步歌词
Meting-API           → 核心中文源，接入 5 个国内平台
                     ├── 网易云音乐
                     ├── QQ 音乐（腾讯）
                     ├── 酷狗音乐
                     ├── 虾米音乐
                     └── 百度音乐
```

**重点**：它没有用任何需要申请 API Key 的服务。全部是公开接口，开箱即用。

搜索的时候，每个 Provider 并发跑自己的，谁先出结果且相似度达标（用 `difflib.SequenceMatcher` 做字符级检验，≥35% 阈值），谁就胜出。如果全部 Provider 都返回低分结果——比如文件名太模糊——就自动选相似度最高那个，同时打日志让人知道。

这个设计对中文歌特别实用。比如你在文件名里写的是「鄭中基 - 無賴」，它能：

1. 先从文件名拆出 `artist=鄭中基, title=無賴`
2. iTunes 搜不到？跳下一个
3. MusicBrainz 也没有？继续
4. Meting（网易云通道）命中 → 返回完整元数据 + 封面 URL
5. 繁体转简体（可选开关）：`鄭中基` → `郑中基`，`無賴` → `无赖`
6. 下载封面，写入标签，保存 LRC 歌词

整个过程不需要你点任何按钮——**真正的一键处理**。

## 那些"藏得很深"的工程细节

我写这个项目的时候有一些偏执，可能比同类工具多走了几步。挑几个有意思的说：

### 1. 速率控制策略（各 API 的亲和力不同）

这是 MusicBrainz 官方要求，不是随便调的：

| Provider | 并发 | 频率 |
|----------|------|------|
| iTunes Search | ✅ 2 并发 | 无延迟 |
| LRCLIB | ✅ 3 并发 | 无延迟 |
| MusicBrainz | ❌ 只能 1 个 | **每秒最多 1 次**（违反会封 IP） |
| Meting-API(公开) | ❌ 1 个 | 每次间隔 500ms~1s |
| Meting-API(自建) | ✅ 2 并发 | 间隔 200ms |

而且限流是每个 Provider 独立互不阻塞的——MusicBrainz 被限速不会拖慢 iTunes 的搜索。碰到 429（Too Many Requests）会自动按指数退避 + 60 秒起等，不崩不丢数据。

### 2. 文件名解析的各种"野路子"命名

这些年我从各种地方收集来的歌，命名方式五花八门：

```
周杰伦 - 晴天.mp3              ← 歌手 - 歌曲
01. 陈奕迅_富士山下.flac       ← 序号_歌手_歌曲
124.这首歌唱给你 王奕心&郑东.mp3 ← 序号.歌曲 歌手（空格分隔）
林俊杰/1008/江南.mp3            ← 来自文件夹的信息
周传雄-寂寞沙洲冷-酷狗.mp3     ← 带"来源后缀"
```

文件名解析器（`filename_parser.py`）用一段不算长但逻辑严密的代码，处理了：

- `-` / `–` / `—` 等多种短横分隔
- 纯空格分隔时用 `rsplit` 取末段 + `_looks_like_artist()` 判断（包含 `&`、命中艺人库、中文姓氏等启发式）
- 多层目录中自动利用文件夹名作为歌手/专辑线索
- 去除了 `- 酷狗`、`- 网易云音乐` 这类"下载后缀"

### 3. 断点续传

处理 300G 的文件不可能一次跑完。`batch_state.json` 逐文件记录状态，中间随时可以停——当前文件允许完成，其余标记 pending。下次启动时可以选择"继续上次的处理"。

ETA 预估也有，基于已处理文件的平均耗时外推剩余时间，数据不足时显示"计算中…"。

### 4. 重复文件检测与冲突解决

同一个歌在不同目录出现了不同版本（比如 128kbps mp3 和 flac），整理到同一个目标目录时怎么办？

它有 **5 种冲突模式**：

- 保留第一个
- 保留最后一个
- **保留音质最好的**（自动对比比特率/采样率/位数/格式，加权评分）
- 自动重命名加序号
- 每次询问

还支持 SHA-256 哈希检测**真正内容完全相同的文件**（比如不同目录下同一首歌的完全一样拷贝），可以选择跳过/覆盖/保留全部/询问。

### 5. 编码修复（中文乱码的根源）

很多老 MP3 的 ID3 标签编码是乱的——标签声称是 ISO-8859-1，实际是 GBK 编码的中文。读出来就是 `ÎÞ·¨` 这种鬼东西。

它的做法是：先通过 Latin-1 把乱码还原为原始字节，然后按用户指定的源字符集（UTF-8/GBK/GB18030/ISO-8859-1）重新解码，再统一转为目标编码。这个逻辑在 `EncodingService` 里，是我试了好几种方案后最终确定能覆盖最多乱码场景的做法。

### 6. 繁体→简体转换

可选的 T2S 转换（依赖纯 Python 库 `hanziconv`），让 `鄭中基` 和 `费翔` 等繁体写法在标签和去重中保持一致。不开的话保持原样，不影响性能。

## 桌面界面：PyQt6 + 深色主题

UI 层用的 PyQt6。不丑，能用，没有赶时髦。

![主界面预览](docs/main-page.png)

布局：

```
┌──────────────────────────────────────────────┐
│  菜单栏（文件/操作/帮助） + 工具栏               │
├─────────────────────┬────────────────────────┤
│  文件列表（左侧）      │  详情面板（右侧）        │
│  ┌──┬────┬────┬────┐ │  封面预览               │
│  │☑│ # │歌手│标题│…│ │  歌手 / 标题 / 专辑     │
│  │☑│ 1 │ ...    │ │  年份 / 流派 / 曲目号    │
│  │☑│ 2 │ ...    │ │  歌词编辑                │
│  └──┴────┴────┴────┘ │  🌐 网络搜索信息 按钮    │
├─────────────────────┴────────────────────────┤
│  进度条 + 暂停/停止按钮                         │
├──────────────────────────────────────────────┤
│  状态栏                                        │
└──────────────────────────────────────────────┘
```

支持的功能：

- **拖放添加文件/文件夹**
- **勾选列**：批处理只处理勾选的文件（默认全勾）
- **序号列**：随勾选/增删自动重排
- **列宽可拖拽调整**
- **文件详情面板**：所有字段可直接编辑，点击保存才写入文件
- **手动搜索**：点「🌐 网络搜索信息」查看多条搜索结果，◀ ▶ 切换预览，确认后填入
- **所有开关运行时生效**：设置里的"是否进行网络搜索"、"是否文件整理"、"是否按模板重命名"等，改完即时生效

### 试听弹窗

双击文件会弹出一个试听窗口：

- 基于 `QMediaPlayer` 的音频播放
- LRC 歌词同步滚动
- **可编辑歌词时间轴**：调整后点击「💾 保存编辑」写入文件标签和同目录 LRC
- 非模态——可以一边听歌一边操作主界面

### Web 管理界面（v1.1.0 新增）

从 v1.1.0 开始，软件启动时可选运行一个 Flask + Waitress 的局域网 Web 服务。手机浏览器打开即可：

- 查看批处理实时状态
- 控制开始/暂停/停止
- 浏览音频目录、加载文件
- 查看实时日志（SSE 推送）

UI 也是深色主题的单页应用。适合把 NAS 上的处理任务丢到后台，然后在沙发上用手机监工。

## 为什么推荐它而不是其他工具？

我不是说它比 MP3Tag 或 MusicBrainz Picard 更"强大"。但在某些场景下，它就是更合适的：

| 场景 | Picard | MP3Tag | AudioFileManager |
|------|--------|--------|-----------------|
| 纯中文歌库搜元数据 | 弱 | 需自己配源 | ✅ 内置网易/QQ/酷狗 |
| 一键批量流水线 | ❌ 需手动确认 | ✅ 但需写脚本 | ✅ 拖入即跑 |
| LRC 歌词同步编辑 | ❌ | ❌ | ✅ |
| 断点续传 | ❌ | ❌ | ✅ |
| 编码修复 | 有限 | 需插件 | ✅ 系统级 |
| 深色主题 | ❌ | ❌ | ✅ |
| Web 远程管理 | ❌ | ❌ | ✅（v1.1.0+） |

**这张表的结论**：如果你跟我一样——中文歌为主、NAS 上有大量散装文件、想要一次拖进去全部搞定的体验——那这个工具就是为你写的。

## 如何上手

安装简单得过分：

```bash
pip install PyQt6 mutagen httpx Pillow chardet
python main.py
```

Windows 用户也可以直接双击 `start.bat`。

依赖就这 5 个，没有花哨的深度学习框架、没有数据库、没有 Docker 容器。

支持 Python 3.8+，在 Windows 上表现最好（因为我就是 Windows 用户，`QMediaPlayer` 在 Linux 上的行为差异还没充分测试）。
